🔑 HOW IT WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Two Access Levels:

  1. PUBLIC (Frontend with Anon Key)
     └─ Can READ: public pages, visible items, available menu
     └─ Cannot WRITE: (protected by RLS)
     └─ Cannot DELETE: (protected by RLS)

  2. ADMIN (Backend with Service Role Key)
     └─ Can READ: everything
     └─ Can CREATE: everything
     └─ Can UPDATE: everything
     └─ Can DELETE: everything
     └─ Uses API routes (not direct DB access)
     └─ Service role key server-side only (secure!)

🔐 SECURITY DETAILS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  How Admin Edit Works:
  
  1. Admin clicks "Edit" in dashboard
     ↓
  2. Frontend sends request to /api/menu/edit
     ↓
  3. Backend API route receives request
     ↓
  4. Backend verifies admin authentication
     ↓
  5. Backend uses SERVICE ROLE KEY
     ↓
  6. Service role bypasses RLS (has full access)
     ↓
  7. Database updates the record
     ↓
  8. Response sent back to frontend
     ↓
  9. Dashboard shows success message ✅

  Why This is Secure:
  • Service role key NEVER sent to frontend
  • Only exists in .env.local (server-side)
  • RLS still protects against unauthorized access
  • All operations verified by API
  • Everything is logged and auditable
